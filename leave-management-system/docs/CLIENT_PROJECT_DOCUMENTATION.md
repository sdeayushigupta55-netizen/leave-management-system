# Police Leave & Operations System  
**Overview for decision-makers and staff**  
*(Plain language — no technical background required)*

**What this document is:** A short description of what the system does, who it is for, and what to plan before wider use.

---

## What problem does this solve?

Police leave today often depends on paper, phone calls, and personal follow-up. That makes it hard to know **where a request sits**, **who must act next**, and **whether rules by rank and duration were followed**.

This system brings leave requests **onto one digital platform** so officers can apply from anywhere they have internet, seniors can approve or forward in order, and everyone can see **status and history** in one place.

---

## What can people do with it? (In everyday words)

| What you get | What it means for your organisation |
|----------------|--------------------------------------|
| **Apply for leave online** | Officers fill a clear form: dates, type of leave, reason. The system works out how many days and which rules apply. |
| **Follow the official chain** | Constable and Head Constable requests move through **SHO/SO → CO → SP → SSP** as appropriate. Senior officers’ requests follow the correct shorter paths (for example toward **SP** and **SSP**). Who can finally approve depends on **rank** and **how long** the leave is. |
| **See where your file is** | Applicants see whether leave is pending, approved, rejected, or forwarded—and **who** has it now. |
| **Less waiting when someone is slow** | If the person who must act does nothing for a set time, the system can **move the request to the next officer** automatically and **record that** in the history. *(The exact waiting time can be set to match your policy—for example one full day.)* |
| **Download approval proof** | Approved leave can be saved or printed as a **PDF certificate** for records. |
| **Work in English or Hindi** | The screens can be used in **English or Hindi** so more staff are comfortable using them. |
| **Beat Book support** | For **Constable** and **Head Constable**, the system can hold **beat-related information** (area, villages, key contacts) alongside their profile. |
| **Reference material** | **BNS 2023** can be opened inside the system as a reference document where you choose to allow it. |
| **Admin control** | A central **Administrator** can add and manage users, and **turn off** accounts that should no longer log in. |

---

## Who uses the system?

- **All police ranks** from Constable up to SSP each get a **home screen suited to their role**—the right shortcuts and the right access (for example, only seniors see **pending leave** from juniors that need action).
- **Administrator** manages users and overall access. **SSP** can be given the same high-level view as the administrator for oversight.

**Ranks covered:** Constable, Head Constable, SI, Inspector, SHO/SO, CO, SP, SSP.

---

## How does a typical leave request work? (Simple steps)

1. **Sign in** using personnel number (P.No.) and registered contact number, then a **one-time code (OTP)** sent to the phone *(in a full pilot or live rollout, this would come from a real SMS service; demos may use a fixed code for training only)*.  
2. Open **Apply leave**, choose dates and type, and submit.  
3. The system sends the request to the **correct first approver** (for example SHO/SO at the same police station, or CO at the same circle, depending on rules).  
4. Each approver can **approve**, **reject**, or **forward** according to **how many days** the leave is and **their rank**.  
5. The applicant checks **Leave status** anytime—no need to visit each office to ask.  
6. When fully approved, a **PDF** can be downloaded for the officer’s file.

---

## Child Care Leave and special cases

For **Child Care Leave**, the system applies **special rules** (for example minimum length of leave and eligibility). This matches the idea that some leave types must go **straight to the highest level** for decision.

---

## Beat Book (for ground staff)

For **Constable** and **Head Constable**, the system can store **beat book** information: which villages or areas belong to the beat, who is incharge, and backup contact details. That keeps operational detail next to the same profile used for leave.

---

## What you need to use it (organisation view)

- **Internet** on the devices officers and approvers will use (office computers, tablets, or phones with a browser).  
- **One agreed web address** for your organisation (like any other internal website).  
- **Someone responsible** for creating accounts and keeping mobile numbers correct (so OTP and alerts work when you move past demo).  
- **A decision** on how long an officer may wait before a request **auto-forwards** to the next level—this should match **your written standing orders**, not only the software default.

---

## Honest note: demo vs full production

The version you may see in a **demo** sometimes uses a **fixed OTP code** for training, so nobody is blocked by SMS setup. Before **real statewide or unit-wide rollout**, your IT or vendor should:

- Connect **real SMS (or another official login method)** so only the real officer can sign in.  
- Host the system on **secure, government-approved** infrastructure if your rules require it.  
- Align **auto-forward time**, **approval limits by days**, and **who counts as “senior”** with your **current circulars and manuals**.

You do **not** need to understand how the software is built to ask for these items in a tender or work order—they are standard expectations for any serious leave portal.

## Short glossary

| Term | Simple meaning |
|------|----------------|
| **P.No.** | Personnel number — like a staff ID for login. |
| **SHO/SO** | Station-level officer in the approval chain. |
| **CO** | Circle Officer — next level in many chains. |
| **SP / SSP** | Superintendent / Senior Superintendent — senior approval. |
| **Beat Book** | Written (here, digital) record of a beat’s area, villages, and contacts. |
| **OTP** | One-time password — usually a code sent by SMS to prove it is really you logging in. |
